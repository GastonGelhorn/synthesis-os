# Cross-Device Usage

The same session (same workspace) is available on every device: the user logs in and sees the same settings (including LLM API keys), workspace, cards, and conversations as on the Mac where the backend is running.

## How it works

- The backend runs on the Mac and exposes **HTTP on `3939`** (desktop app uses localhost) and **HTTPS on `3940`** (required for Safari/iOS).
- The frontend on another device (iPad, another PC) logs in and calls `GET /api/user/sync-state`, which returns all settings (including API keys) and the saved workspace for that user.
- Those values are applied as session state, so what you see on the iPad is the same as on the Mac.
- Real-time agent events stream via SSE at `GET /api/kernel/events`.

## Quick setup (recommended)

The fastest way to get cross-device working is the diagnostic script, which generates certs, verifies the backend, and updates `.env` automatically:

```bash
bash scripts/check-remote-access.sh
```

Run with `--force-cert` to regenerate certificates (e.g. if your Mac IP changed or the cert expired):

```bash
bash scripts/check-remote-access.sh --force-cert
```

After the script finishes, follow the **iPad certificate install** steps it prints.

## Manual setup

### 1. Generate TLS certificates

Safari on iOS **requires HTTPS** for all fetch requests. Use `openssl` to generate a self-signed certificate that covers your Mac's local IP:

```bash
APP_DATA="$HOME/Library/Application Support/com.synthesis.synthesis-os/tls"
mkdir -p "$APP_DATA"

openssl req -x509 -newkey rsa:2048 \
  -keyout "$APP_DATA/key.pem" \
  -out "$APP_DATA/cert.pem" \
  -days 825 -nodes \
  -subj "/CN=SynthesisOS Dev TLS" \
  -addext "subjectAltName=DNS:localhost,IP:127.0.0.1,IP:<YOUR_MAC_IP>"
```

Replace `<YOUR_MAC_IP>` with your Mac's IP on the local network (find it with `ipconfig getifaddr en0`).

> **Validity:** The certificate is valid for 825 days (iOS rejects certs valid for more than 825 days).

### 2. Configure the frontend

Set `VITE_API_BASE` in `apps/desktop/.env` to point to the HTTPS backend on the Mac:

```env
VITE_API_BASE=https://<YOUR_MAC_IP>:3940
```

### 3. Build and run

```bash
cd apps/desktop
npm run build       # bakes VITE_API_BASE into the frontend
npm run dev:tauri   # starts backend + Tauri app
```

In the terminal you should see:
```
[HTTP] Listening on http://0.0.0.0:3939
[HTTP] Listening on https://0.0.0.0:3940 (TLS)
```

### 4. Install the certificate on iPad (required once)

Safari on iOS will block fetch requests to a server with an untrusted certificate, even if you accepted the browser warning. You must install the cert as a trusted system profile:

1. On **iPad Safari**, navigate to:
   ```
   http://<YOUR_MAC_IP>:3939/api/cert.pem
   ```
   (Use HTTP port 3939 for the download — no cert trust needed.)

2. Tap **Allow** when Safari asks to download the configuration profile.

3. Go to **Settings → General → VPN & Device Management**, tap the **SynthesisOS Dev TLS** profile → **Install** → enter your passcode → **Install** again.

4. Go to **Settings → General → About → Certificate Trust Settings**, find **SynthesisOS Dev TLS**, and toggle **Enable Full Trust**.

5. Open Safari and navigate to `https://<YOUR_MAC_IP>:3940`. The app should load and log in normally.

> If you regenerate the certificate (new IP, expired cert), remove the old profile in **Settings → General → VPN & Device Management** before installing the new one.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| "No se puede analizar la respuesta" | Safari blocks fetch to untrusted HTTPS cert | Complete step 4 above (install cert + enable full trust) |
| "Failed to fetch" / connection refused | HTTPS server not running | Check logs for `[HTTP] Listening on https://...`; run script with `--force-cert` and restart |
| Cert shows dates 1/1/75 – 1/1/96 | Generated by old rcgen code with bad defaults | Run `bash scripts/check-remote-access.sh --force-cert` to regenerate with openssl |
| HTTPS works but page is 404 | `dist/` not built | Run `cd apps/desktop && npm run build` before `npm run dev:tauri` |
| Cert valid but iOS still blocks | Full trust not enabled | Go to **Settings → General → About → Certificate Trust Settings** and enable trust |
| IP changed (different WiFi) | Cert SANs don't cover new IP | Run `bash scripts/check-remote-access.sh --force-cert`, reinstall profile on iPad |

## How certificates are managed

- On app startup, `ensure_tls_certs()` in `lib.rs` checks `~/Library/Application Support/com.synthesis.synthesis-os/tls/` for `cert.pem` and `key.pem`.
- If the files exist and the `sans.txt` fingerprint matches the current local IPs, the existing cert is used as-is.
- If IPs changed or certs are missing, new certs are generated automatically with `rcgen`.
- The diagnostic script generates certs with `openssl` and writes a `sans.txt` fingerprint so the Rust code doesn't overwrite them.
- The HTTPS server (port `3940`) only starts if valid cert files are found. HTTP (port `3939`) always runs.

## Alternative: env var cert paths

You can also point the backend at cert files in any location via environment variables (useful for mkcert or production certs):

```bash
export SYNTHESIS_HTTPS_CERT="/path/to/cert.pem"
export SYNTHESIS_HTTPS_KEY="/path/to/key.pem"
cd apps/desktop && npm run dev:tauri
```
